# Evaluation Bundle

This directory holds the benchmark manifest, per-video labels, reference predictions, and generated results for short evaluation clips.

## Layout

- `benchmark_manifest.json`: manifest for the bundled starter benchmark.
- `benchmark_manifest_public_datasets.json`: curated public-dataset tracking subset built from MOT17 and VisDrone clips.
- `annotations/`: manually maintained ground-truth detections and expected events.
- `predictions/`: detector, tracker, and event outputs. These can be reference JSON files or outputs generated by `python -m scripts.run_benchmark`.
- `predictions_public/`: generated predictions for the public-dataset subset.
- `results/`: generated evaluation summaries.
- `videos/`: short benchmark clips.

## Benchmark target

The repo is now structured to support a more credible benchmark pass with:

- 8 to 15 short clips
- multiple scene types
- explicit occlusion cases
- false-positive traps
- both person and vehicle scenarios

The bundled asset set is still a small starter benchmark to keep the repository light. The manifest schema and runner are designed so you can scale the same workflow to MOT17, VIRAT, VisDrone, or private surveillance footage.

There is now also a real public-dataset subset in the repo derived from MOT17 and VisDrone frame sequences. It is intentionally curated and small enough to version-control.

## Manifest schema

Each manifest entry can include:

```json
{
  "video_id": "office_intrusion_short",
  "video_path": "videos/office_intrusion_short.mp4",
  "ground_truth": "annotations/office_intrusion_short.json",
  "predictions": "predictions/office_intrusion_short.json",
  "scene_types": ["office", "indoor"],
  "challenge_tags": ["restricted_zone"],
  "subject_classes": ["person"],
  "config_override": {
    "zones": []
  }
}
```

Use `config_override` to attach clip-specific zones, enabled event types, and thresholds without creating a separate YAML file per video.

## Annotation and prediction schema

Each annotation or prediction file uses:

```json
{
  "video_id": "office_intrusion_short",
  "fps": 10,
  "duration_seconds": 6.0,
  "detections": [
    {
      "frame_index": 0,
      "track_id": 1,
      "class": "person",
      "bbox": [0, 0, 10, 20],
      "score": 0.96
    }
  ],
  "events": [
    {
      "event_type": "intrusion",
      "zone": "restricted_lab",
      "class": "person",
      "track_id": 1,
      "entry_frame_index": 2,
      "frame_index": 2,
      "match_tolerance_frames": 3
    }
  ],
  "runtime": {
    "device": "cpu",
    "frames_processed": 60,
    "wall_clock_seconds": 4.8,
    "effective_fps": 12.5,
    "stage_total_seconds": {
      "detect": 3.1,
      "track": 0.2,
      "events": 0.1
    }
  }
}
```

## Commands

Generate predictions and evaluate in one pass:

```bash
python -m scripts.run_benchmark --manifest data/eval/benchmark_manifest.json --config configs/default.yaml --device cpu --output-json data/eval/results/latest.json --output-markdown docs/results.md
```

Score an existing set of prediction JSON files:

```bash
python -m scripts.evaluate_events --manifest data/eval/benchmark_manifest.json --output-json data/eval/results/latest.json --output-markdown docs/results.md
```

Run CPU and GPU passes separately if you want direct throughput comparisons.

Build the public-dataset subset locally after downloading MOT17 and VisDrone into `data/external/`:

```bash
python -m scripts.frames_to_video --frames-dir data/external/mot17/train/MOT17-04-FRCNN/img1 --out data/eval/videos/mot17_04_clip.mp4 --fps 30 --start-frame 151 --end-frame 750 --resize-width 960
python -m scripts.import_mot17 --gt data/external/mot17/train/MOT17-04-FRCNN/gt/gt.txt --seqinfo data/external/mot17/train/MOT17-04-FRCNN/seqinfo.ini --out data/eval/annotations/mot17_04_tracking.json --start-frame 151 --end-frame 750 --min-visibility 0.2 --scale-x 0.5 --scale-y 0.5

python -m scripts.frames_to_video --frames-dir data/external/visdrone/VisDrone2019-MOT-train/sequences/uav0000099_02109_v --out data/eval/videos/visdrone_uav0000099_02109_clip.mp4 --fps 30 --start-frame 151 --end-frame 750 --resize-width 960
python -m scripts.import_visdrone_mot --gt data/external/visdrone/VisDrone2019-MOT-train/annotations/uav0000099_02109_v.txt --out data/eval/annotations/visdrone_uav0000099_02109_tracking.json --fps 30 --start-frame 151 --end-frame 750 --scale-x 0.7142857143 --scale-y 0.7142857143
```

Then score the full public-dataset subset:

```bash
python -m scripts.run_benchmark --manifest data/eval/benchmark_manifest_public_datasets.json --config configs/default.yaml --profile edge_cpu --device cpu --model models/yolo11n.pt --predictions-dir data/eval/predictions_public --output-json data/eval/results/public_dataset_cpu.json --output-markdown docs/results_public_dataset_cpu.md
python -m scripts.render_metrics_report --input-json data/eval/results/public_dataset_cpu.json --output-html docs/results_public_dataset_cpu.html
```

To shrink checked-in benchmark videos before rerunning inference:

```bash
python -m scripts.compress_benchmark_videos --input-dir data/eval/videos --output-dir data/eval/videos_compressed --crf 28 --preset slow --max-width 960
```

On Windows shells where `ffmpeg` is installed but not exposed on `PATH`, pass `--ffmpeg-bin <path-to-ffmpeg.exe>`.

If you re-encode any checked-in benchmark clip, rerun the benchmark and refresh the reports before committing so the prediction JSON files still match the shipped videos.

Expected interpretation of the current checked-in public-dataset result:

- It is a lightweight CPU baseline using the `edge_cpu` profile.
- It is not tuned specifically for MOT17 or VisDrone.
- It mixes crowded street tracking and moving-camera aerial tracking, so weak raw MOTA/IDF1 should be read as baseline stress numbers, not final system quality.

## Manifest explanation

`benchmark_manifest.json` is the single source of truth for what the benchmark means.

Each entry tells the evaluator:

- which clip is in scope
- which ground-truth file defines expected detections and events
- which prediction file should be scored
- which scene and challenge tags the clip contributes
- which per-clip config overrides were used, if any

If you add, remove, or retag clips in the manifest, the benchmark has changed and the reported totals should be regenerated.

## Reproducibility notes

The checked-in `docs/results.md` and `data/eval/results/latest.json` files are score-only artifacts produced from the checked-in reference prediction JSON files. That means:

- the exact reproduction commands are deterministic
- hardware is not part of the checked-in result provenance
- model runtime only becomes relevant when running `python -m scripts.run_benchmark`

For live runs, report all of the following with the results:

- command used
- manifest path
- config path
- model path
- device used
- hardware used

## Labeling guidance

- Expand each benchmark pass to 8 to 15 short clips before presenting metrics externally.
- Keep at least one clip each for occlusion, crowding, and false-positive traps.
- Include both person and vehicle footage if the repo claims both.
- For intrusion and loitering, mark `entry_frame_index` and the expected `frame_index` for the alert.
- Keep `match_tolerance_frames` tight enough to catch late or early alerts.
